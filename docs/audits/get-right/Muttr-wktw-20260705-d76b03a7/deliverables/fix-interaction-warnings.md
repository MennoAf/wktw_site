# Fix Interaction Warnings

*w.k.t.w. — Fix Coordination Guide for WKTW*

> **Coordination Required.** Our review found that the proposed fixes interact with each other in ways that require coordinated implementation. Shipping these fixes independently — without the coordination described below — risks regressions, conflicts, and wasted effort.

---

## Cross-Tier Conflicts

These fixes operate at different tiers (Server, Platform, Template) but modify the same systems. Implementing them independently will cause conflicts.

1. CSP TRIPLICATION — CRITICAL SEQUENCING CONFLICT: Three separate proposals independently author the same CSP header: 'prescan-escalation-csp-confirmed-absent' (server tier), 'escalation-resolution-csp' (server tier), and 'det-security-headers-https-weknowthewhy-com-about' (platform tier). All three target the same Netlify _headers / netlify.toml [[headers]] /* path. If all three are executed, the last writer wins and the other two are silently discarded. Netlify applies only one Content-Security-Policy value per path match — duplicate header keys in _headers are not merged. The platform proposal (det-security-headers) adds 'unsafe-inline' as a Phase 1 interim, while both server proposals use SHA-256 hashes without 'unsafe-inline'. If the platform proposal deploys first and the server proposals deploy second, the stricter hash-based policy wins — but if the platform proposal deploys last, 'unsafe-inline' survives in production indefinitely, defeating the entire hardening exercise. Resolution: designate exactly one proposal as the canonical CSP owner (recommend 'escalation-resolution-csp' as it is the most complete), mark the other two as superseded, and ensure the final policy is hash-based with no 'unsafe-inline'.

2. CONSENT BANNER QUINTUPLE IMPLEMENTATION CONFLICT: Five proposals independently implement a full consent banner replacement: 'privacy-consent-banner-absent' (platform), 'consent-banner-not-rendering' (platform), 'prescan-escalation-console-errors' (platform), 'ghost-2-consent-banner-hidden-interactive' (template), and 'ux-mobile-consent-banner-viewport-impact' (template). Each creates a different src/components/ConsentBanner.astro with different internal architectures — some use localStorage, some use cookies, some use a CSS class toggle, some use inline onclick. If more than one is executed, the last import in BaseLayout.astro wins and the others are dead code, but the conflicting cookie/localStorage key names ('consent-decision' vs 'consent_state' vs 'site_consent') mean a user who accepted consent under one implementation will be shown the banner again after a redeploy using a different key. Additionally, 'privacy-1-no-pre-consent-cookies' (platform) implements a third consent architecture via window.__consent.gateScript(), which is incompatible with the ConsentBanner component model used by all five other proposals. Resolution: designate one canonical implementation (recommend 'ghost-2-consent-banner-hidden-interactive' as it is the most architecturally complete), standardize on a single storage key, and mark all others as superseded.

3. PLAUSIBLE PROXY CONFLICT — DUPLICATE NETLIFY REWRITE RULES: 'analytics-no-ga4-no-conversion-tracking' (platform) and 'ux-analytics-client-side-only-data-loss' (platform) both add Netlify redirect rules proxying /js/script.js → https://plausible.io/js/script.js and /api/event → https://plausible.io/api/event. If both are merged into netlify.toml, Netlify will have duplicate redirect rules for the same source paths. Netlify applies the first matching redirect rule and ignores subsequent duplicates — this is silent, not an error, but creates maintenance confusion. Additionally, 'js-plausible-only-third-party' (platform) instructs developers to keep the Plausible script at its direct plausible.io URL and explicitly documents why SRI is incompatible, while the proxy proposals change the script src to a first-party path — these are architecturally contradictory. Resolution: decide whether Plausible runs via first-party proxy (analytics/ux-analytics proposals) or direct (js-plausible-only-third-party), implement one path only, and remove the conflicting proposal.

4. FONT SCOPING CONFLICT — JETBRAINS MONO: 'font-face-overload-3-families' (platform), 'js-2-jetbrains-mono-font-loaded-unnecessarily' (template), and 'resource-loading-font-strategy-good' (template) all independently implement the same fix: moving JetBrains Mono @font-face out of global CSS and into a scoped CodeBlock/MonospaceFonts component. All three propose creating different component files (CodeBlock.astro vs MonospaceFonts.astro) with slightly different scoping strategies. If more than one executes, the @font-face declaration will exist in multiple component files, causing it to be emitted into every page that imports any of those components — defeating the optimization. Resolution: designate one proposal as the canonical implementation, pick one component name, and mark the others superseded.

5. CACHE HEADER CONFLICT — NETLIFY.TOML VS _HEADERS FILE: 'server-4-cache-control-revalidate-every-request' (server) authors cache rules in public/_headers, while 'server-transport-cdn-cache-status-missing' (server) authors the same rules in netlify.toml [[headers]] blocks. Both target /_astro/* and /fonts/* with identical Cache-Control values. When both files exist, Netlify merges them but the precedence order is: netlify.toml [[headers]] takes priority over _headers file for the same path. This means the _headers file rules are silently shadowed. The proposals are not wrong individually, but deploying both creates a maintenance trap where engineers editing _headers believe they are changing cache behavior when they are not. Resolution: pick one mechanism (recommend netlify.toml for all cache and security headers — single source of truth), remove the _headers approach, and update all dependent proposals accordingly.

6. CONTACT FORM COMPONENT CONFLICT — TRIPLE IMPLEMENTATION: 'escalation-6-console-errors-unverifiable' (platform), 'prescan-override-1-contact-form-severity' (template), and 'conversion-no-error-recovery-evidence' (template) all independently create a new contact form component with fetch-based submission, isSubmitting guard, error surfacing, and field preservation. Each proposes a different component file path (src/components/ContactForm.astro vs src/components/ContactCTA.astro vs inline in src/pages/contact.astro). The form-name values differ across proposals, which would cause Netlify Forms to register multiple distinct form endpoints for what should be one form. Resolution: designate one proposal as the canonical contact form implementation, standardize the form-name value, and ensure all other proposals that reference the contact form defer to it.

7. INLINE CTA COMPONENT CONFLICT — DUPLICATE COMPONENT DEFINITIONS: 'ux-conversion-no-form-on-page' (template) and 'ux-conversion-01' (template) both create src/components/InlineCTA.astro with different prop interfaces. 'ux-conversion-no-form-on-page' creates a form-bearing component (name + email + message fields), while 'ux-conversion-01' creates a pure anchor element with no form. These are fundamentally different components with the same filename. Additionally, 'ux-conversion-01' appears twice in the template proposals list with identical content — a direct duplicate entry. Resolution: reconcile the two InlineCTA designs into one component with a variant prop distinguishing anchor-only from form-bearing, remove the duplicate entry for ux-conversion-01.

8. GCM V2 CONSENT DEFAULT PLACEMENT CONFLICT: 'consent-banner-not-rendering' (platform) instructs developers to move the gtag('consent', 'default', {...}) call INTO the ConsentBanner component's inline script, while 'prescan-escalation-console-errors' (platform) instructs developers to keep it in <head> BEFORE the ConsentBanner component. These are mutually exclusive placement strategies. If the default fires inside the banner component, it executes after the GTM/GA4 script tag — violating Google's requirement that consent defaults fire before any tag. Resolution: the default MUST remain in <head> before the GTM script tag (prescan-escalation-console-errors is correct on this point); update consent-banner-not-rendering to remove the instruction to relocate it.

## Coherence Issues

These aren't direct conflicts, but the fixes make inconsistent assumptions or use incompatible configurations. Left unresolved, they'll produce unpredictable behavior.

1. HSTS DECLARED IN TWO PLACES: 'server-transport-no-redirect-chains-pass' (server) adds HSTS to a Netlify _headers file, while 'server-transport-http2-tls13-pass' (server) adds HSTS to netlify.toml [[headers]]. Both target /*. Given the _headers vs netlify.toml precedence conflict noted above, one of these will silently win. The HSTS max-age values are not specified in either proposal, creating a risk that different engineers set different values (e.g., 31536000 vs 63072200) depending on which proposal they read. Standardize on netlify.toml with an explicit max-age=31536000; includeSubDomains value.

2. ALT-SVC HEADER ADVISORY: 'server-transport-http2-tls13-pass' (server) adds an Alt-Svc header advertising h3 and h3-29. Netlify's edge network manages QUIC/HTTP3 negotiation at the platform layer — injecting a user-authored Alt-Svc header may conflict with or duplicate Netlify's own Alt-Svc emission, potentially advertising a port or protocol version that Netlify's edge does not actually support on this plan tier. This is advisory but should be verified against Netlify's current HTTP/3 support documentation before deploying.

3. VARY: ACCEPT-ENCODING HEADER TRUNCATION: 'server-transport-cdn-cache-status-missing' (server) includes a Vary: Accept-Encoding instruction that is cut off mid-sentence in the how field ('Add a Vary: Accept-Encoding header to all blocks. Netlify's edge uses this to store separate cache entries for Brotli-compressed and g'). The proposal is incomplete as written. The developer cannot implement this step from the truncated text. This must be completed before implementation.

4. CONSENT STORAGE KEY INCONSISTENCY ACROSS ALL PROPOSALS: Across the six consent banner proposals, four different localStorage/cookie keys are used: 'consent-decision' (privacy-consent-banner-absent), 'consent_state' (privacy-1-no-pre-consent-cookies), 'site_consent' cookie (ghost-2-consent-banner-hidden-interactive), and an unspecified key in consent-banner-not-rendering. A user who visits the site after sequential deploys using different keys will be shown the consent banner repeatedly, creating a GDPR compliance regression (re-asking for consent already given) and a poor UX. All proposals must converge on one key name before any is deployed.

5. PLAUSIBLE DEFER VS ASYNC INCONSISTENCY: 'js-plausible-only-third-party' (platform) instructs switching from async to defer for the Plausible script tag. 'ux-analytics-client-side-only-data-loss' (platform) instructs removing async and adding defer. These agree on the outcome but the first-party proxy proposals change the script src simultaneously, meaning the defer/async attribute change and the src change must be coordinated in one commit — not applied independently. If the src changes without the defer change, or vice versa, an intermediate state ships to production.

6. NETLIFY FORMS FORM-NAME VALUE PROLIFERATION: Across the template proposals, the following form-name values appear: 'contact' (conversion-no-error-recovery-evidence, form-1-missing-label), 'inline-contact' and 'inline-micro' (ux-conversion-no-form-on-page), 'service-page-cta', 'proof-page-cta', 'about-page-cta', 'article-page-cta' (prescan-override-1-contact-form-severity). Netlify Forms charges per submission and has plan-based form limits. Proliferating form endpoints without a deliberate strategy may exhaust form quotas or create a dashboard that is unmanageable. This is not a blocking issue but requires a deliberate naming convention decision before implementation.

7. BREADCRUMB JSON-LD OWNERSHIP CONFLICT: 'ux-mobile-2' (template) moves BreadcrumbList JSON-LD into a LegalBreadcrumb.astro component, while 'escalation-review-2-unknown-schema' (template) centralizes all JSON-LD through a SchemaOrg.astro component fed by typed builder functions. If both are implemented, BreadcrumbList will be emitted twice on legal pages — once from LegalBreadcrumb and once from SchemaOrg — causing duplicate structured data that Google's Rich Results validator flags as an error. Resolution: SchemaOrg.astro must be the sole JSON-LD emitter; LegalBreadcrumb must call the schema builder and pass the result to SchemaOrg rather than emitting its own script tag.

8. CONTENT-VISIBILITY: AUTO CONTRADICTORY GUIDANCE: 'stability-2-content-visibility-not-needed' (template) explicitly closes the content-visibility finding as a true negative and states no implementation is required. However, 'visual-stability-cls-zero-pass' (template) step 4 instructs using CSS containment (contain: layout) on injection targets for consent overlays and banners. contain: layout is a subset of content-visibility: auto's containment model. While not directly contradictory, the two proposals use containment terminology inconsistently and a developer reading both may apply contain: layout globally (regressing the stability-2 decision) or not at all (missing the CLS guard in visual-stability). Clarify that contain: layout in visual-stability-cls-zero-pass is scoped only to JS-injected overlay containers, not applied globally.

9. SKIP LINK FINDING SCOPE MISMATCH: 'prescan-escalation-skip-link-truncation' (template) proposes fixing the audit pipeline's internal signature serialization — it is a fix to the audit tooling, not to the live site. This proposal does not belong in the template tier (which contains site code changes) and should not be deployed as a site change. It is an internal pipeline maintenance task. Including it in the template proposals alongside site code changes risks a developer treating it as a site deployment task.

10. FONT PRELOAD CROSSORIGIN ATTRIBUTE — DUPLICATE VERIFICATION TASKS: 'a11y-font-display-swap-pass' (template) step 1 and 'resource-loading-font-subsetting' (template) step 5 both independently instruct the developer to verify that <link rel='preload' as='font'> declarations carry crossorigin='anonymous'. This is the same verification task duplicated across two proposals. Not a conflict, but creates redundant work. One proposal should own this verification and the other should reference it.

## Within-Tier Interactions

Fixes within the same tier that share implementation surfaces (DOM manipulation, CSS properties, MutationObservers) and need coordinated rollout.

### Platform Tier

**Proposals:** `det-security-headers-https-weknowthewhy-com-about`, `analytics-no-ga4-no-conversion-tracking`, `privacy-consent-banner-absent`, `privacy-2-pre-consent-plausible-fires`, `ux-analytics-client-side-only-data-loss`, `consent-banner-not-rendering`, `prescan-escalation-console-errors`, `escalation-6-console-errors-unverifiable`, `privacy-1-no-pre-consent-cookies`, `font-face-overload-3-families`, `backwards-compat-astro-static-pass`, `resource-6-css-not-render-blocking-concern`, `js-minimal-payload-pass`, `js-plausible-only-third-party`, `content-to-code-ratio-css-heavy`, `ghost-markup-no-orphaned-assets`, `ghost-1-no-js-injection-pass`

**Conflicts:**

- CRITICAL — CONSENT BANNER QUADRUPLICATION: Four proposals independently implement a full consent banner component (privacy-consent-banner-absent, privacy-2-pre-consent-plausible-fires, consent-banner-not-rendering, prescan-escalation-console-errors). Each creates a different src/components/ConsentBanner.astro with incompatible architectures: (1) privacy-consent-banner-absent uses inline styles + localStorage key 'consent-decision' + data-consent-action attributes; (2) privacy-2-pre-consent-plausible-fires uses a separate consent-store.ts module + CustomElement ConsentGatedScript + localStorage key 'site_consent_decisions'; (3) consent-banner-not-rendering uses is:inline script + localStorage key 'cookie_consent' + cookie-based persistence; (4) prescan-escalation-console-errors uses cookie-based persistence (document.cookie) + localStorage key 'user_consent' + event-driven footer trigger. Deploying more than one will produce duplicate banner DOM, conflicting localStorage/cookie keys, and multiple competing gtag('consent','default') calls. Only one implementation can ship. The team must select exactly one.
- CRITICAL — PLAUSIBLE LOADING STRATEGY TRIPLICATION: Three proposals each independently modify how Plausible loads, with mutually exclusive approaches: (1) analytics-no-ga4-no-conversion-tracking proxies via /js/script.js and /api/event with defer; (2) ux-analytics-client-side-only-data-loss proxies via /stats/js/script.js and /stats/api/event with defer; (3) privacy-2-pre-consent-plausible-fires gates Plausible behind a ConsentGatedScript custom element that injects the script only after consent. The proxy paths in (1) and (2) are different (/js/ vs /stats/js/) — deploying both netlify.toml redirect blocks creates two active proxy routes pointing to the same upstream, wasting redirect evaluation. More critically, (3) gates Plausible behind consent while (1) and (2) load it unconditionally — these are architecturally contradictory consent postures. The team must decide: is Plausible consent-gated or consent-exempt? That decision determines which loading proposal is valid.
- CRITICAL — CSP HEADER DUPLICATION WITH CONFLICTING VALUES: Two proposals independently write a Content-Security-Policy header to netlify.toml [[headers]] for '/*': det-security-headers-https-weknowthewhy-com-about (Phase 1 CSP with 'unsafe-inline' in script-src, plausible.io in script-src and connect-src) and js-plausible-only-third-party (CSP without 'unsafe-inline', with report-uri). Netlify merges duplicate header keys by last-write-wins within a single [[headers]] block, but if they are separate [[headers]] blocks for the same path, behavior is undefined and implementation-dependent. The two CSP strings are substantively different — one permits 'unsafe-inline', the other does not. Deploying both produces an unpredictable effective CSP. These must be reconciled into a single [[headers]] block with one CSP value.
- CONFLICT — CONSENT STORAGE KEY FRAGMENTATION: Across the four consent banner proposals, four different localStorage/cookie keys are used for the same semantic value (user consent decision): 'consent-decision' (privacy-consent-banner-absent), 'site_consent_decisions' (privacy-2-pre-consent-plausible-fires), 'cookie_consent' (consent-banner-not-rendering), 'user_consent' (prescan-escalation-console-errors), and 'consent_state' (privacy-1-no-pre-consent-cookies). If any two of these proposals are partially deployed (e.g., old banner removed, new banner added), returning visitors whose consent was stored under the old key will be re-prompted unnecessarily. A migration shim is required if the key changes.
- CONFLICT — GOOGLE CONSENT MODE DEFAULT PLACEMENT: consent-banner-not-rendering moves the gtag('consent','default') call INTO the ConsentBanner component's is:inline script, while prescan-escalation-console-errors places it in BaseLayout.astro's <head> as a separate inline script BEFORE the ConsentBanner component. These are mutually exclusive placement strategies. If both are partially applied, the default call may execute twice (harmless but wasteful) or in the wrong order relative to the GTM script tag (harmful — defaults set after GTM loads are ignored by Google's tag infrastructure).
- CONFLICT — NETLIFY TOML REDIRECT PATH COLLISION: analytics-no-ga4-no-conversion-tracking defines redirects for /js/script.js and /api/event. ux-analytics-client-side-only-data-loss defines redirects for /stats/js/script.js and /stats/api/event. The BaseLayout script tag in ux-analytics-client-side-only-data-loss uses src='/stats/js/script.js' while analytics-no-ga4-no-conversion-tracking uses src='/js/script.js'. If both redirect blocks are present but only one script tag src is updated, the other proxy route is dead weight. If both script tags are somehow present, Plausible loads twice.
- CONFLICT — FONT SCOPE PROPOSALS ASSUME DIFFERENT STARTING STATES: font-face-overload-3-families assumes JetBrains Mono @font-face is currently in a global stylesheet and proposes moving it to CodeBlock.astro. backwards-compat-astro-static-pass assumes @font-face declarations already exist somewhere (global or component) and proposes adding woff fallbacks to them. If font-face-overload-3-families ships first and moves @font-face into CodeBlock.astro, backwards-compat-astro-static-pass must target CodeBlock.astro — not the global stylesheet. If backwards-compat-astro-static-pass ships first, font-face-overload-3-families must move the already-updated (woff-inclusive) declarations. The proposals must be sequenced explicitly.
- CONFLICT — JS BUDGET ENFORCER AND CONSENT BANNER JS: js-minimal-payload-pass sets a JS_BUDGET_BYTES threshold of 51,200 bytes (50KB) against the current ~29KB baseline. The consent banner proposals (privacy-2-pre-consent-plausible-fires in particular) introduce a ConsentGatedScript custom element with a non-trivial class definition, a consent-store.ts module, and multiple Astro component scripts. The analytics instrumentation proposal (analytics-no-ga4-no-conversion-tracking) adds analytics.ts, ContactFormTracker.astro, and GlobalEventTracking.astro scripts. Combined, these additions may push the uncompressed JS bundle past the 50KB threshold, causing the budget enforcer to fail the build immediately after the other proposals are deployed. The threshold must be re-evaluated after all JS-adding proposals are scoped.

**Interaction effects:**

- INTERACTION — CSP PHASE 1 BLOCKS CONSENT BANNER INLINE SCRIPTS: det-security-headers-https-weknowthewhy-com-about Phase 1 CSP includes 'unsafe-inline' in script-src, which permits the inline scripts used by consent-banner-not-rendering (is:inline) and prescan-escalation-console-errors (is:inline IIFE). However, Phase 2 CSP removes 'unsafe-inline' and replaces it with SHA-256 hashes. Any consent banner that uses is:inline scripts must have its script content hashed and added to the Phase 2 CSP script-src. The CI guard in det-security-headers (check-inline-scripts.mjs) will flag these inline scripts as violations when PHASE_2_ACTIVE=true. The consent banner implementation chosen must either: (a) use Astro's non-inline <script> tags (which Astro bundles externally), or (b) have its hash pre-computed and registered in the Phase 2 CSP. This dependency is not documented in either proposal.
- INTERACTION — PLAUSIBLE PROXY CHANGES CSP CONNECT-SRC REQUIREMENT: If analytics-no-ga4-no-conversion-tracking or ux-analytics-client-side-only-data-loss ships the first-party proxy, the CSP connect-src directive in det-security-headers-https-weknowthewhy-com-about must be updated. The current Phase 1 CSP allowlists 'https://plausible.io' in connect-src. After proxying, the event POST goes to the site's own origin (e.g., /api/event or /stats/api/event), which is covered by 'self' — so plausible.io can be removed from connect-src. Failing to update this leaves an unnecessary external origin in the allowlist, weakening the CSP's specificity.
- INTERACTION — CONSENT GATE AFFECTS PLAUSIBLE PROXY UTILITY: If privacy-2-pre-consent-plausible-fires gates Plausible behind consent, the first-party proxy (analytics-no-ga4-no-conversion-tracking, ux-analytics-client-side-only-data-loss) still provides value for consenting users — but the proxy's primary stated benefit (defeating ad blockers) is partially undermined because the script only loads after a consent interaction, by which point the page has already rendered. The proxy is still worth deploying for consented return visitors, but the framing in both proxy proposals (defeating blockers on initial load) does not account for the consent gate delay.
- INTERACTION — SERVER-SIDE FORM LOG AND CONSENT: ux-analytics-client-side-only-data-loss creates a Netlify Function that logs form submissions including UTM attribution. This server-side log fires regardless of consent state. Under GDPR, server-side logging of IP address (via context.geo) and referrer URL constitutes personal data processing. If the consent banner proposals establish a consent-first architecture, the server-side log may need to be scoped to exclude geo data for non-consenting users, or justified under a separate legal basis (legitimate interest for fraud prevention/spam filtering). This interaction is not addressed in either proposal.
- INTERACTION — GHOST-1 ZERO-HYDRATION ASSERTION AND CONSENT BANNER: ghost-1-no-js-injection-pass asserts zero Astro island markers on static templates. The consent banner proposals all use <script> blocks (some is:inline, some standard Astro <script>). Standard Astro <script> tags in components do NOT produce <astro-island> markers — they are bundled as regular script chunks. However, if any consent banner component uses client:* directives (none currently do, but this is a future risk), the zero-hydration assertion would catch it. The interaction is benign today but the assertion provides a useful safety net against future consent banner refactors that accidentally introduce hydration.
- INTERACTION — JS ISLAND AUDIT AND ANALYTICS INSTRUMENTATION: js-minimal-payload-plus creates an island-audit integration that scans for client:* directives. analytics-no-ga4-no-conversion-tracking adds ContactFormTracker.astro and GlobalEventTracking.astro with standard <script> blocks (not client:* directives). These will NOT be flagged by the island audit. However, the JS budget enforcer in js-minimal-payload-pass will count their bundled output bytes. The analytics proposal's scripts are non-trivial (typed event system, event delegation, queue mechanism) and will contribute measurably to the JS bundle. This is expected and acceptable, but the budget threshold should be updated in the same PR.
- INTERACTION — FONT SCOPE CHANGE AND CSS COVERAGE AUDIT: font-face-overload-3-families moves @font-face declarations from global CSS into CodeBlock.astro's scoped <style>. ghost-markup-no-orphaned-assets adds a Playwright CSS coverage audit. After the font move, the global stylesheet will no longer contain @font-face declarations, which will change the coverage profile for non-CodeBlock pages (previously the @font-face bytes were 'unused' on those pages; after the move they are absent entirely). The coverage audit's baseline must be re-established after the font move ships — running the audit against the pre-move state and comparing to post-move will show a false improvement if the baseline is not reset.
- INTERACTION — MULTIPLE NETLIFY.TOML MODIFICATIONS: Five proposals independently modify netlify.toml (det-security-headers, analytics-no-ga4-no-conversion-tracking, ux-analytics-client-side-only-data-loss, js-minimal-payload-pass, ghost-markup-no-orphaned-assets). Each adds [[headers]], [[redirects]], or [build] blocks. Netlify.toml does not support merging — all changes must be manually reconciled into a single file. Specifically: the [[headers]] blocks from det-security-headers and js-plausible-only-third-party must be merged into one block; the [[redirects]] blocks from analytics-no-ga4-no-conversion-tracking and ux-analytics-client-side-only-data-loss must be reconciled to use one proxy path; the [build] command from ghost-markup-no-orphaned-assets, ghost-1-no-js-injection-pass, and escalation-6-console-errors-unverifiable must be chained into a single command string.
- INTERACTION — PRIVACY-1 CONSENT GATE AND ALL OTHER CONSENT PROPOSALS: privacy-1-no-pre-consent-cookies implements a window.__consent global API as the canonical gating mechanism. If this proposal ships, all other consent banner proposals become redundant — their banner DOM and localStorage logic would conflict with the __consent API's own state management. Conversely, if any of the other four consent proposals ships instead, privacy-1's __consent API is orphaned. These five proposals are mutually exclusive at the architectural level, not just the implementation level.

**Recommended approach:** The 17 proposals decompose into four functional domains that should each be resolved as a single coordinated implementation rather than independent fixes:

DOMAIN 1 — CONSENT INFRASTRUCTURE (5 proposals: privacy-consent-banner-absent, privacy-2-pre-consent-plausible-fires, consent-banner-not-rendering, prescan-escalation-console-errors, privacy-1-no-pre-consent-cookies): These five proposals all solve the same root problem — a broken consent banner — with five different architectures. The team should select exactly one implementation. Recommended: privacy-2-pre-consent-plausible-fires provides the most composable architecture (consent-store.ts as single source of truth, ConsentGatedScript custom element for declarative gating, clean separation of concerns). However, its ConsentBanner component uses the CSS `hidden` attribute for initial state (banner starts hidden, shown by JS), which means JS failure leaves the banner invisible — the opposite of the legally safe default. The correct architecture combines: (a) privacy-consent-banner-absent's inline-styles-visible-by-default approach for the banner HTML, (b) privacy-2-pre-consent-plausible-fires's consent-store.ts module for state management, (c) consent-banner-not-rendering's is:inline script placement for synchronous GCM default initialization, and (d) prescan-escalation-console-errors's event-driven footer trigger pattern. This synthesis should be a single new proposal replacing all five.

DOMAIN 2 — ANALYTICS LOADING AND INSTRUMENTATION (3 proposals: analytics-no-ga4-no-conversion-tracking, ux-analytics-client-side-only-data-loss, js-plausible-only-third-party): These three proposals all modify how Plausible loads and what it tracks. The higher-order solution: (a) decide the consent posture for Plausible (consent-gated vs. legitimate-interest exempt) — this is a legal decision, not a technical one, and it gates everything else; (b) implement one proxy path (pick /js/script.js or /stats/js/script.js, not both); (c) implement the analytics.ts typed event wrapper once (analytics-no-ga4-no-conversion-tracking has the cleanest implementation); (d) implement the server-side form log once (ux-analytics-client-side-only-data-loss). The SRI documentation from js-plausible-only-third-party is advisory and can be a code comment, not a separate proposal.

DOMAIN 3 — SECURITY HEADERS AND CSP (2 proposals: det-security-headers-https-weknowthewhy-com-about, js-plausible-only-third-party): Both write to the same netlify.toml [[headers]] block. The higher-order solution is a single [[headers]] block that incorporates: the phased CSP from det-security-headers (Phase 1 with 'unsafe-inline', Phase 2 with hashes), the Permissions-Policy from js-plausible-only-third-party, and the HSTS/X-Content-Type-Options/X-Frame-Options/Referrer-Policy from both. **Conflicts:**

- CRITICAL — Duplicate CSP proposals: 'prescan-escalation-csp-confirmed-absent' and 'escalation-resolution-csp' both propose adding a Content-Security-Policy header to public/_headers (and netlify.toml as an alternative). One must be selected and the other discarded before implementation.
- CRITICAL — _headers vs netlify.toml split across four proposals: 'prescan-escalation-csp-confirmed-absent' and 'escalation-resolution-csp' target public/_headers; 'server-transport-cdn-cache-status-missing', 'server-transport-no-brotli', 'server-1-ttfb-pass', and 'server-transport-http2-tls13-pass' target netlify.toml; 'server-4-cache-control-revalidate-every-request' targets public/_headers with a netlify.toml alternative; 'server-transport-no-redirect-chains-pass' targets public/_headers. If both files are deployed with overlapping path matchers (e.g., '/*' or '/_astro/*'), the netlify.toml Cache-Control rules in 'server-transport-cdn-cache-status-missing', 'server-transport-no-brotli', and 'server-1-ttfb-pass' will be silently overridden by the _headers rules from 'server-4-cache-control-revalidate-every-request' and 'server-transport-no-redirect-chains-pass'. - DIRECT CONFLICT — HTML Cache-Control TTL disagreement between proposals targeting the same path: 'server-transport-cdn-cache-status-missing' sets s-maxage=3600 (1 hour) for '/*' HTML; 'server-1-ttfb-pass' sets s-maxage=60 (1 minute) for '/*.html' and '/'; 'server-transport-no-brotli' sets Cache-Control: public, no-cache for '/*.html' and '/'. - DIRECT CONFLICT — HSTS max-age disagreement: 'server-transport-no-redirect-chains-pass' sets max-age=31536000 (1 year) without 'preload'; 'server-transport-http2-tls13-pass' sets max-age=63072000 (2 years) with 'preload'. - OVERLAP — 'server-transport-no-brotli' netlify.toml block and 'server-transport-cdn-cache-status-missing' netlify.toml block both define [[headers]] rules for '/_astro/*', '/fonts/*' (or '/assets/*'), '/*.html', and '/'. ### General

- CACHE IMMUTABILITY + CONSENT BANNER REDEPLOY REGRESSION: The server-tier cache proposals ('server-4-cache-control-revalidate-every-request', 'server-transport-cdn-cache-status-missing') set Cache-Control: public, max-age=31536000, immutable on /_astro/* assets. The consent banner proposals generate a new ConsentBanner.astro component whose compiled JS and CSS will land in /_astro/ with a content hash. If the consent banner is redeployed with a bug fix (e.g., correcting the storage key name), the new asset gets a new hash and a new URL — correct behavior. However, any user whose browser cached the OLD consent banner JS under the old hash will continue running the old (broken) banner until their cache expires in one year, because the HTML page (served with max-age=0, must-revalidate) will reference the NEW hash while the old file is unreachable. This is the correct immutable caching behavior, but it means consent banner bugs have a one-year tail on cached clients unless the HTML revalidation cycle is working correctly. Mitigation: verify HTML pages are NOT cached with immutable (the proposals correctly exempt /* from immutable caching), and confirm Netlify's CDN invalidates HTML on every deploy.
- CSP HASH INVALIDATION ON EVERY ASTRO BUILD: The server-tier CSP proposals ('prescan-escalation-csp-confirmed-absent', 'escalation-resolution-csp') use SHA-256 hashes for inline scripts. Astro's build pipeline (via Vite) may modify inline script content between builds — hydration scripts, hoisted component scripts, and Vite's module preload polyfill are all candidates for content changes that invalidate hashes. If the CSP is deployed with hardcoded hashes and a subsequent Astro build changes any inline script (e.g., a dependency update changes Vite's injected preload script), the site will silently break — scripts will be blocked by CSP with no visible error to the developer unless they check the browser console. The proposals acknowledge this risk and recommend a postbuild hash-generation script, but if that automation is not implemented before the static CSP is deployed, there is a production breakage window. Mitigation: the hash automation script must be implemented and verified before the CSP header goes live; do not deploy the CSP with manually-computed hashes.
- PLAUSIBLE PROXY + CSP SCRIPT-SRC CONFLICT: The analytics proxy proposals ('analytics-no-ga4-no-conversion-tracking', 'ux-analytics-client-side-only-data-loss') change the Plausible script src from https://plausible.io/js/script.js to a first-party path (e.g., /js/script.js). The CSP proposals allowlist https://plausible.io in script-src. After the proxy is deployed, the script loads from 'self' (same origin), which is already permitted by script-src 'self'. However, the connect-src directive in the CSP proposals explicitly allowlists https://plausible.io for the event beacon POST. After the proxy is deployed, the beacon also routes through the first-party proxy (/api/event), so connect-src 'self' covers it — but if the CSP is deployed before the proxy, the connect-src must include https://plausible.io. If the proxy is deployed before the CSP, connect-src needs only 'self'. Deploying these in the wrong order will either block analytics beacons (CSP first, no proxy) or leave an unnecessary external allowlist entry (proxy first, CSP second). Mitigation: deploy the proxy and the CSP in the same commit, with connect-src including both 'self' and https://plausible.io as a belt-and-suspenders measure.
- CONSENT GATE + PLAUSIBLE PROXY ANALYTICS BLACKOUT: 'privacy-2-pre-consent-plausible-fires' (platform) gates Plausible behind consent, loading it only after user acceptance. 'ux-analytics-client-side-only-data-loss' (platform) routes Plausible through a first-party proxy. If the consent gate dynamically injects a <script src='https://plausible.io/js/script.js'> after consent (the direct URL), but the proxy has been deployed and the CSP only allows 'self' for script-src, the dynamically injected script will be blocked by CSP. The consent gate must inject the proxied URL (/js/script.js), not the direct plausible.io URL. These three proposals (consent gate + proxy + CSP) must be implemented as a coordinated unit with the proxied URL used consistently throughout.
- IMMUTABLE CACHE HEADERS + FONT SCOPING REFACTOR: The font scoping proposals ('font-face-overload-3-families', 'js-2-jetbrains-mono-font-loaded-unnecessarily', 'resource-loading-font-strategy-good') move JetBrains Mono WOFF2 files from global CSS to a scoped component. The WOFF2 files themselves live in /fonts/ or are referenced from /_astro/ CSS. After the refactor, the CSS file that previously triggered the WOFF2 download no longer exists (it has a new content hash). Any user whose browser cached the old CSS under the old hash will have a stale CSS file that references a WOFF2 path that may no longer exist at the same URL if the font file path changed. Since WOFF2 files in /fonts/ are served with max-age=31536000, immutable, a cached WOFF2 at the old path will persist in the browser cache even if the file is removed from the server. This is benign (the old CSS and old WOFF2 are a consistent pair), but if the WOFF2 path changes during the refactor, the old CSS will reference a 404. Mitigation: keep WOFF2 files at stable /fonts/ paths using absolute URLs (as recommended in the proposals), never rename or move the font files.
- TAILWIND PURGE FIX + EXISTING COMPONENT STYLES REGRESSION: 'content-to-code-ratio-css-heavy' (platform) proposes fixing Tailwind content glob patterns to enable proper tree-shaking. If purge was previously broken and the full Tailwind utility set was shipping, some components may have been relying on utility classes that were present in the bundle despite not being explicitly referenced in source files (e.g., dynamically constructed class names like 'text-' + color, or classes injected by third-party libraries). Fixing the purge config will remove these classes from the bundle, potentially breaking component styles that were accidentally working. Mitigation: after fixing the purge config, do a full visual regression pass across all page templates before deploying to production.
- HEADING HIERARCHY FIXES + SCHEMA BREADCRUMB REGRESSION: Multiple template proposals fix heading hierarchy (h1→h3 skip on /contact/, 'det-wcag-improper-content-structure-https-weknowthewhy-com-contact'; site-wide heading governance, 'a11y-heading-hierarchy-pass'). Simultaneously, 'escalation-review-2-unknown-schema' (template) rebuilds BreadcrumbList JSON-LD. If heading elements are used as the source of breadcrumb label text (a pattern some Astro templates use), changing heading levels or text content will silently change breadcrumb labels in the structured data. Mitigation: after any heading change, re-run the schema validation script proposed in escalation-review-2-unknown-schema to confirm BreadcrumbList labels still match the visible page hierarchy.
- NETLIFY FORMS PROLIFERATION + ANALYTICS CONVERSION TRACKING REGRESSION: 'prescan-override-1-contact-form-severity' (template) creates multiple form endpoints with unique form-name values per page type. 'analytics-no-ga4-no-conversion-tracking' (platform) instruments the contact form with a 'Contact Form Submitted' Plausible custom event. If the form-name values change (from 'contact' to 'service-page-cta', etc.), the Plausible event tracking code that fires on form submission must be updated to fire for all new form variants — otherwise conversion events will only be tracked for the original 'contact' form and the new distributed forms will be invisible in analytics. Mitigation: the Plausible event instrumentation must be applied to all form variants, not just the original contact form.
- INLINE CTA FORMS + NETLIFY STATIC FORM DETECTION: 'ux-conversion-no-form-on-page' (template) and 'prescan-override-1-contact-form-severity' (template) add inline forms via Astro components. Netlify's form detection bot scans static HTML at deploy time. If these forms are rendered inside Astro components that are conditionally imported (e.g., only on certain page templates), Netlify's bot must encounter the rendered HTML — not just the component source. If any of these forms are rendered only on client-side (e.g., inside a client:load island), Netlify's bot will not detect them and form submissions will return 404. The proposals correctly note that forms must be in static HTML, but this constraint must be enforced at code review time for every new form addition.
- S-MAXAGE CDN CACHING + CONSENT BANNER VISIBILITY REGRESSION: 'server-transport-cdn-cache-status-missing' (server) sets s-maxage=3600 on HTML documents, allowing Netlify's CDN to cache HTML pages for up to one hour. The consent banner proposals render the banner as visible-by-default static HTML. If a user visits a CDN-cached HTML page, the banner HTML is present. However, if the consent banner's localStorage check (which hides the banner for returning visitors) fails to execute (e.g., due to a JS error), the banner will be visible on every CDN-cached page load for returning visitors who already consented. This is the correct failure mode (show banner rather than hide it), but it means a JS error in the consent banner creates a persistent UX regression for all users hitting the CDN cache until the cache expires. Mitigation: the inline synchronous script that hides the banner for returning visitors must be extremely minimal and defensively written — no external dependencies, no async operations, wrapped in try-catch.
- DOM QUALITY PROPOSAL CSP CONFLICT: 'dom-quality-excellent' (template) step 3 proposes adding style-src 'self' (no 'unsafe-inline') to the CSP to block inline styles. However, multiple other template proposals use inline styles for critical rendering: 'privacy-consent-banner-absent' uses inline styles to make the banner visible without JS, 'ux-mobile-consent-banner-viewport-impact' uses inline styles for the banner layout, and 'a11y-2-touch-targets-undersized' may generate inline style attributes via JavaScript. A CSP with style-src 'self' and no 'unsafe-inline' will block all of these inline styles, breaking the consent banner's no-JS-required visibility guarantee and potentially breaking touch target enforcement. The server-tier CSP proposals include style-src 'self' 'unsafe-inline' — which conflicts with dom-quality-excellent's stricter style-src 'self'. These must be reconciled: either accept 'unsafe-inline' for styles (weaker security, broader compatibility) or audit and eliminate all inline style usage before deploying the strict style-src.

## Coordination Guidance

Recommendations from the review for sequencing and coordinating these fixes.

**Cross-tier:** The proposal set is not approved for simultaneous deployment as written. The cross-tier conflicts are not theoretical — they will cause production failures if the proposals are executed concurrently or in an uncoordinated sequence. The three critical blockers before any deployment begins: (1) Designate exactly one canonical consent banner implementation and one canonical storage key — the current six competing implementations will cause consent state loss on redeploy. (2) Designate exactly one canonical CSP header owner and resolve the 'unsafe-inline' vs hash-based policy conflict — deploying both will result in the weaker policy surviving. (3) Resolve the Plausible proxy vs direct-URL decision before implementing either the CSP or the consent gate — the three are tightly coupled and must deploy as a coordinated unit. Once these three conflicts are resolved, the remaining coherence issues (font scoping duplication, form-name proliferation, breadcrumb JSON-LD ownership) are sequencing and naming decisions that can be resolved at implementation time without production risk. The individual proposals are technically sound in isolation — the problem is exclusively cross-tier coordination, which is expected at this proposal volume and is resolvable with a brief implementation sequencing session before work begins.

**Platform tier:** Approval is blocked by the five-way consent banner conflict and the three-way Plausible loading conflict. These are not implementation details — they represent contradictory architectural decisions that cannot coexist in the same codebase. Deploying any subset of the conflicting proposals without explicit reconciliation will produce: duplicate ConsentBanner components overwriting each other in src/components/, conflicting localStorage keys causing returning visitors to be re-prompted, multiple gtag('consent','default') calls with potentially different wait_for_update values, and two active Plausible proxy routes in netlify.toml pointing to the same upstream. The individual proposals are technically sound in isolation — the problem is that five engineers independently solved the same problem without coordination. The resolution path is: (1) hold a 30-minute architectural decision meeting to select one consent implementation and one Plausible loading strategy; (2) produce a single reconciled proposal for each domain as described in higher_order_solution; (3) re-submit for holistic review. The non-conflicting proposals (font-face-overload-3-families, backwards-compat-astro-static-pass, resource-6-css-not-render-blocking-concern, content-to-code-ratio-css-heavy, escalation-6-console-errors-unverifiable) can be approved and shipped independently while the consent/analytics conflict is resolved, provided netlify.toml modifications are manually merged.

**Server tier:** The proposals are individually well-reasoned but were authored independently and cannot be deployed as a batch without conflicts. The two CSP proposals are a hard duplicate — one must be closed before any work begins.
