# Cluster Deep Dive: CSP Absence — No Script Injection Protection

**Cluster ID:** cluster_010  
**Architectural Pattern:** Server-side configuration  
**Findings in cluster:** 3  
**Systemic issue:** Yes  
**Compliance implications:** GDPR  

---

## 1. The Big Picture

Every page on weknowthewhy.com is currently served without a Content-Security-Policy header. In practical terms, this means the browser has no instructions from the server about which scripts are permitted to run on the page. If an attacker were able to inject a malicious script — through a compromised dependency, a supply-chain attack on a third-party resource, or any other injection vector — the browser would execute it without question. There is no browser-enforced boundary preventing that script from reading form inputs, harvesting data entered into the contact form, or exfiltrating it to an external server. For a site whose primary conversion mechanism is a contact form, that is a meaningful exposure.

The practical user-facing risk is not a degraded experience in the ordinary sense — visitors will not notice the absence of a CSP header. The risk is that the absence creates a silent attack surface. The contact form at /contact/ is the site's sole conversion path. Any data a visitor submits through that form — name, email, message content — is processed in a browser environment that has no policy-level protection against injected scripts reading and forwarding that data. The site's other security headers (HSTS, X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy) are all present and correctly configured. CSP is the single gap in an otherwise complete security header posture.

The reason this matters beyond the technical is regulatory. The contact form collects personal data from visitors who are likely to include EU residents. Under GDPR, the controller — WKTW — is responsible for implementing appropriate technical measures to protect that personal data. A missing CSP header, in the context of a form that collects personal data, is a measurable gap in those technical measures. The three findings in this cluster all confirm the same underlying fact from different angles of the audit: CSP is definitively absent across every page of the site.

---

## 2. The Root Cause

The root cause is a single omission in the Netlify deployment configuration. Netlify does not add a Content-Security-Policy header by default — it must be explicitly declared in either a `_headers` file in the project's `public/` directory or in a `[[headers]]` block in `netlify.toml`. The audit confirmed that neither configuration exists. The evidence is unambiguous: the server response headers were enumerated across multiple pages, and the seven security-relevant headers present (HSTS, X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy, and others) were all explicitly listed. CSP was not among them. The ten additional headers omitted from the raw data were confirmed to be standard Netlify operational headers — request IDs, caching metadata, content-type, content-length — not security headers. The absence of CSP is not an ambiguity in the data; it is a confirmed gap.

The complicating factor that has likely deferred this configuration is the presence of 7 inline `<script>` blocks in the site's HTML output. A strict CSP that disallows inline scripts would break those scripts immediately, which makes a naive deployment risky. This is a solvable problem — either through `'unsafe-inline'` as a transitional measure, or through SHA-256 hashes for each inline script block — but it does mean the fix requires a small amount of upfront inventory work before deployment. The single external script dependency (Plausible Analytics at `plausible.io`) is already well-scoped and straightforward to allowlist.

## 3. Each Finding

### Finding: Missing recommended security headers
**ID:** det-security-headers-https-weknowthewhy-com-about  
**Severity:** Medium | **Tier:** Platform | **Legal liability:** Yes

**What's broken:** The deterministic header scan confirmed that Content-Security-Policy is absent from server responses across five measured pages: the homepage, /about, /contact, /insights/why-most-audits-dont-change-anything, and /legal/privacy. The finding was generated by a direct inspection of response headers, not inference. The remediation data explicitly lists CSP as the missing header alongside a note that it mitigates XSS and data injection attacks.

**Why it matters:** The /contact page is the site's primary conversion mechanism — the only path through which a visitor can initiate a business relationship with WKTW. Without a CSP, the browser enforces no policy against injected scripts executing on that page. A script injected through any vector (compromised CDN, dependency confusion, or future third-party addition) could silently read and exfiltrate form field contents before submission. This directly threatens the integrity of the contact_form KPI: not by reducing form completions, but by creating conditions under which completed submissions could be intercepted. The /legal/privacy page being affected is also notable — a privacy policy page served without script injection protection carries its own reputational dimension.

**The fix:** Add a `[[headers]]` block to `netlify.toml` declaring the CSP directive for all paths (`/*`). The proposed policy — `default-src 'self'; script-src 'self' https://plausible.io 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self'; font-src 'self'; connect-src 'self' https://plausible.io; frame-ancestors 'none'; base-uri 'self'; form-action 'self'` — covers the site's current script surface including the 7 inline scripts and the Plausible external dependency. The `'unsafe-inline'` directive is a transitional measure; the longer-term path is to replace it with per-script SHA-256 hashes or to migrate inline scripts to external files.

### Finding: Escalation review — CSP confirmed absent (prescan-2-1 upheld)
**ID:** prescan-escalation-csp-confirmed-absent  
**Severity:** High | **Tier:** Server | **Legal liability:** Yes

**What's broken:** This finding documents the audit's internal escalation process, which challenged an earlier prescan result to ensure the CSP absence was not a data artifact. The escalation examined whether the "10 omitted headers" in the raw response data could have included a CSP header that was simply not displayed. The verdict was upheld: the omitted headers were confirmed to be Netlify operational and caching headers (request IDs, server-timing, age, date, etag, content-type, content-length), not security headers. Security headers are consistently enumerated in full in Netlify response data. CSP is absent.

**Why it matters:** The escalation finding elevates the severity to High because it closes the last reasonable alternative explanation. This is not a measurement ambiguity or a tooling gap — it is a confirmed architectural omission. The finding also surfaces the specific inline script count (7) as a material constraint on the fix approach. A CSP deployed without accounting for those 7 inline scripts would break site functionality, which is why the two-phase approach (permissive first, tightened second) is the recommended path rather than a single strict deployment.

**The fix:** The proposed implementation approach from this finding is the most detailed of the three: enumerate all inline `<script>` blocks across the built output in `dist/`, generate SHA-256 hashes for each unique script body using `openssl dgst -sha256`, and include those hashes in the `script-src` directive. This eliminates the need for `'unsafe-inline'` entirely and produces a policy that is both functional and strict. The effort is higher than the Phase 1 permissive deployment but represents the correct end state.

### Finding: Escalation resolution — CSP absence confirmed with high confidence
**ID:** escalation-resolution-csp  
**Severity:** High | **Tier:** Server | **Legal liability:** Yes

**What's broken:** This finding is the final resolution of the escalation chain, confirming CSP absence with high confidence after two rounds of review. It adds no new technical evidence beyond the prior two findings but serves as the authoritative close of the audit's internal verification process. The reasoning is consistent: the 7 listed security headers in the response data represent a complete enumeration of security-relevant headers; CSP would logically appear alongside HSTS, X-Content-Type-Options, X-Frame-Options, Referrer-Policy, and Permissions-Policy if it were present; it does not appear; therefore it is absent.

**Why it matters:** Three independent audit passes — a deterministic header scan, a prescan escalation, and an escalation resolution — all arrive at the same conclusion. This convergence removes any residual uncertainty about whether the finding warrants action. For a client considering whether to prioritise this fix, the answer is unambiguous: the absence is real, it is site-wide, and it affects every page including the contact form.

**The fix:** The implementation path recommended here — creating or updating `public/_headers` in the Astro project (which Astro copies to `dist/` at build time) — is the most Astro-native approach and avoids modifying `netlify.toml` if that file is managed separately. Either file location achieves the same result at the Netlify edge. The policy scope should target `/*` to ensure coverage across all current and future pages without per-path maintenance.

## 4. The Unified Fix Strategy

All three findings resolve through a single configuration change. The coordinated approach has two phases, ordered by risk and effort.

**Phase 1 — Deploy permissive CSP immediately (estimated effort: quick win)**

This is a 5-minute configuration change that closes the zero-coverage gap. Add the following to `netlify.toml`:

```toml
[[headers]]
  for = "/*"
  [headers.values]
    Content-Security-Policy = "default-src 'self'; script-src 'self' https://plausible.io 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self'; font-src 'self'; connect-src 'self' https://plausible.io; frame-ancestors 'none'; base-uri 'self'; form-action 'self'"
```

Alternatively, create `public/_headers` in the Astro project root:

```
/*
  Content-Security-Policy: default-src 'self'; script-src 'self' https://plausible.io 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self'; font-src 'self'; connect-src 'self' https://plausible.io; frame-ancestors 'none'; base-uri 'self'; form-action 'self'
```

This policy is permissive on inline scripts (`'unsafe-inline'`) but it is categorically better than no CSP. It immediately restricts external script sources to `self` and `plausible.io`, blocks framing from external origins (`frame-ancestors 'none'`), restricts form submissions to same-origin (`form-action 'self'`), and prevents base tag injection (`base-uri 'self'`). Deploy this in the next release cycle.

**Phase 2 — Tighten the policy by eliminating `'unsafe-inline'` (estimated effort: medium)**

This phase requires inventorying the 7 inline scripts in the built HTML output. Run the following against the Astro build output:

```bash
find dist -name '*.html' | xargs grep -oP '(?s)(?<=<script>).*?(?=</script>)' | sort -u
```

For each unique inline script body, generate a SHA-256 hash:

```bash
echo -n 'SCRIPT_CONTENT' | openssl dgst -sha256 -binary | openssl base64
```

Replace `'unsafe-inline'` in the `script-src` directive with the collected hashes: `'sha256-HASH1' 'sha256-HASH2'` etc. Alternatively — and preferably — migrate the inline scripts to external `.js` files referenced with `<script src="...">`, which eliminates the hashing requirement entirely and produces a cleaner policy. Astro's component model makes this straightforward for most inline script patterns.

**Priority rationale:** Phase 1 should not wait for Phase 2. The compliance and security exposure exists today. A permissive-but-present CSP is a meaningful improvement over no CSP, and it can be deployed without touching any application code. Phase 2 is the correct long-term posture but should not block Phase 1.

### Legal Exposure — Plain Language

GDPR Article 25 (Data Protection by Design and by Default) and Article 32 (Security of Processing) require controllers to implement appropriate technical measures to protect personal data. The contact form at /contact/ collects personal data — at minimum, name and email address — from visitors who may be EU residents. The absence of a CSP header means the browser has no server-enforced policy preventing injected scripts from reading that form data before or during submission. This is a measurable gap in the technical measures WKTW is obligated to maintain. In the event of a data breach involving the contact form, the absence of a CSP — particularly given that all other security headers are present — would be a visible omission in any regulatory review of the technical safeguards in place. The fix is a single configuration line. The risk-to-effort ratio makes this one of the clearest remediation cases in the audit.
