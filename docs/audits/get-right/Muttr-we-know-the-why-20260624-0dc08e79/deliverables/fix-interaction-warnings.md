# Fix Interaction Warnings

*w.k.t.w. — Fix Coordination Guide for We Know the Why*

> **Coordination Required.** Our review found that the proposed fixes interact with each other in ways that require coordinated implementation. Shipping these fixes independently — without the coordination described below — risks regressions, conflicts, and wasted effort.

---

## Cross-Tier Conflicts

These fixes operate at different tiers (Server, Platform, Template) but modify the same systems. Implementing them independently will cause conflicts.

1. ANALYTICS ARCHITECTURE FORK — UNRESOLVED DECISION REQUIRED BEFORE DEPLOY: Platform tier contains two mutually exclusive analytics strategies that cannot both be implemented. 'resource-loading-dual-analytics-redundancy' and 'attribution-split-pixel-not-applicable' (Phase 3) advocate removing GTM entirely and retaining Plausible as the sole analytics platform. All other analytics proposals ('ux-analytics-triple-redundancy-kpi-corruption', 'an-2-duplicate-analytics-gtm-ga4-plausible', 'js-4-ga4-collect-aborted-data-loss', 'js-unused-bytes-low-but-present', 'escalation-5-ga4-err-aborted-root-cause', 'ux-analytics-cross-domain-unknown') assume GTM is retained as the sole GA4 orchestrator. Template proposal 'ux-analytics-form-submit-untracked' is written entirely around GTM dataLayer pushes and a GTM GA4 Event tag — it becomes inert if GTM is removed. This is a client decision gate, not a technical one. The entire analytics tier is blocked until the client chooses: (A) GTM-only GA4 + optional Plausible, or (B) Plausible-only. All proposals must be conditionally scoped to the chosen path before any deploy.

2. CSP NONCE IMPLEMENTATION SURFACE CONFLICT — SERVER VS PLATFORM TIER: Server proposals 'escalation-1-sri-gtm-ga-mitigation' and 'prescan-escalation-sri-gtm-incompatibility' describe nonce generation as occurring at the server/CDN layer (Next.js middleware, Express, Nginx lua_by_lua_block, Cloudflare Worker). Platform proposal 'det-security-headers-https-weknowthewhy-com-about' specifies a Netlify Edge Function (csp-nonce.ts) as the nonce generation and HTML-rewriting mechanism. These are not the same implementation surface. The Netlify Edge Function approach is the correct one for this stack (Astro SSG on Netlify) — the server-tier proposals reference server runtimes that do not exist in this deployment. Only one implementation surface can own nonce injection. The Edge Function approach must be designated the canonical implementation and the server-tier descriptions treated as illustrative only, not as parallel instructions.

3. FONT SELF-HOSTING OWNERSHIP CONFLICT — PLATFORM VS TEMPLATE TIER: Platform proposals 'fonts-google-fonts-privacy-performance' and 'resource-3-google-fonts-external-render-blocking-risk' both specify downloading Inter and Lora variable WOFF2 files and inlining @font-face declarations in <head>. Template proposal 'resource-loading-font-subsetting-opportunity' covers the same scope with additional subsetting detail (pyftsubset commands, Latin unicode ranges). All three propose placing files in a versioned static asset directory and removing the Google Fonts <link> tags. These proposals are duplicates targeting the same files, the same template location, and the same cache headers. Implementing all three independently risks triple-writing the @font-face block and producing conflicting file paths. One proposal must be designated the implementation owner; the others must be explicitly marked superseded. Recommendation: 'resource-loading-font-subsetting-opportunity' is the most operationally complete (includes subsetting commands and size targets) and should own the implementation.

4. CONSENT BANNER IMPLEMENTATION CONFLICT — FOUR PROPOSALS, THREE INCOMPATIBLE ARCHITECTURES: Platform proposals 'privacy-1-no-consent-mechanism', 'det-gdpr-pre-consent-tracking-https-weknowthewhy-com', 'privacy-cookies-consent-banner-dark-pattern', and 'ux-interactive-consent-banner-button-clarity' all implement a consent gate, but with structurally different approaches. 'privacy-1-no-consent-mechanism' uses a dynamically injected banner (JS-rendered). 'det-gdpr-pre-consent-tracking-https-weknowthewhy-com' requires the banner to be server-rendered in raw HTML with display:block inline style. 'privacy-cookies-consent-banner-dark-pattern' uses <template> tags to wrap inert script elements. 'ux-interactive-consent-banner-button-clarity' implements a state machine with a role='dialog' ARIA pattern. These cannot all be implemented simultaneously — they conflict on DOM structure, script-blocking mechanism, and banner mount strategy. A single consent implementation must be designed that satisfies all four sets of requirements (GDPR gating, visual parity, ARIA dialog, state machine). Template proposal 'consent-banner-ux-on-privacy-page' adds route-suppression logic that must be layered on top of whichever architecture is chosen.

5. TOUCH TARGET REMEDIATION OVERLAP — THREE PROPOSALS TARGETING THE SAME ELEMENTS: Platform proposals 'ux-mobile-hamburger-discoverability' and 'mobile-nav-touch-targets-ux' both target the hamburger button (40px → 48px) and primary CTA. Template proposals 'accessibility-touch-targets-undersized', 'det-wcag-improper-content-structure-https-weknowthewhy-com-about', and 'ux-mobile-nav-cta-undersized' target overlapping element sets. 'accessibility-touch-targets-undersized' proposes a new touch-target-floors.css file with six scoped rulesets. 'mobile-nav-touch-targets-ux' proposes a Tailwind base-layer rule. These two CSS strategies are architecturally incompatible — a standalone CSS file and a Tailwind base-layer rule will produce specificity conflicts on the same elements. One approach must be chosen. Given the Astro/Tailwind stack, the Tailwind base-layer approach in 'mobile-nav-touch-targets-ux' is the correct surface; 'accessibility-touch-targets-undersized' should be superseded.

6. MOBILE NAV ARIA REMEDIATION OVERLAP — THREE PROPOSALS TARGETING THE SAME COMPONENT: Platform proposal 'ux-mobile-hamburger-discoverability' and template proposals 'a11y-5-mobile-menu-duplicate-nav-no-aria-label', 'a11y-mobile-menu-nav-no-aria-label', 'det-wcag-missing-name-role-value-https-weknowthewhy-com-contact', and 'ghost-markup-mobile-menu-hidden-links' all write aria-label, aria-expanded, and aria-hidden attributes to the same hamburger button and mobile nav elements. If implemented independently, the last deploy wins and intermediate states may be inconsistent. These must be merged into a single header component change ticket.

7. HEADING HIERARCHY REMEDIATION OVERLAP — TWO PROPOSALS, SAME COMPONENT: Template proposals 'det-wcag-improper-content-structure-https-weknowthewhy-com-about', 'a11y-heading-hierarchy-skip', and 'ux-heading-hierarchy-skip' all propose a polymorphic Heading Astro component with a level prop. All three describe the same component (Heading.astro), the same prop interface, and the same call-site refactor. These are duplicates. One must be designated the implementation owner.

## Coherence Issues

These aren't direct conflicts, but the fixes make inconsistent assumptions or use incompatible configurations. Left unresolved, they'll produce unpredictable behavior.

1. CACHE HEADER PROPOSALS ARE TRIPLICATED IN SERVER TIER: 'server-transport-no-cdn-cache-headers', 'st-6-cache-control-no-immutable-on-hashed-assets', and 'cache-control-no-caching-static-assets' all produce the same output: a _headers file or netlify.toml [[headers]] block applying Cache-Control: public, max-age=31536000, immutable to /_astro/*. They differ only in minor details (Vary: Accept-Encoding in one, CDN-Cache-Control in another, X-Cache-Debug in a third). Implementing all three independently will produce duplicate or conflicting header rules. These must be merged into a single implementation ticket that incorporates all additive details: immutable on /_astro/*, Vary: Accept-Encoding, Netlify-CDN-Cache-Control, and the staging-only X-Cache-Debug guard.

2. STANDALONE GTAG.JS REMOVAL IS SPECIFIED IN NINE SEPARATE PROPOSALS: 'escalation-1-sri-gtm-ga-mitigation' (Phase 1), 'prescan-escalation-sri-gtm-incompatibility' (Step 1), 'ux-analytics-triple-redundancy-kpi-corruption' (Phase 1), 'resource-loading-dual-analytics-redundancy' (Step 2-3), 'an-2-duplicate-analytics-gtm-ga4-plausible' (Phase 2), 'js-4-ga4-collect-aborted-data-loss', 'js-unused-bytes-low-but-present' (Phase 2), 'escalation-5-ga4-err-aborted-root-cause' (Step 3), and 'attribution-split-pixel-not-applicable' (Phase 2) all instruct the developer to remove the same two script blocks from the same template location. This is a single two-line HTML deletion. It must be executed exactly once. The risk of nine proposals all containing this instruction is that a developer implements it nine times across nine tickets, or that the instruction is treated as already done after the first ticket and skipped in subsequent ones — leaving dependent steps in those proposals unexecuted.

3. GOOGLE FONTS REMOVAL IS SPECIFIED IN FIVE SEPARATE PROPOSALS: 'fonts-google-fonts-privacy-performance', 'resource-3-google-fonts-external-render-blocking-risk', 'resource-loading-font-subsetting-opportunity', 'privacy-1-no-consent-mechanism' (Step 1), and 'det-gdpr-pre-consent-tracking-https-weknowthewhy-com' (implied by consent gating) all instruct removal of the Google Fonts <link> tags and preconnect hints. This is a single template change. Same single-execution risk as the gtag.js removal above.

4. CSP REPORT-ONLY → ENFORCE PROMOTION LACKS A CROSS-PROPOSAL HANDOFF PROTOCOL: Server proposals describe a phased CSP rollout (report-only first, then enforce). Platform proposal 'det-security-headers-https-weknowthewhy-com-about' also describes this two-phase approach. Neither tier specifies who owns the promotion decision, what violation threshold triggers promotion, or how the report-uri endpoint is shared between tiers. Without a named owner and a defined promotion criterion, the CSP will remain in report-only mode indefinitely.

5. PLAUSIBLE CONSENT GATING IS INCONSISTENT ACROSS PROPOSALS: 'privacy-1-no-consent-mechanism' gates Plausible behind consent. 'resource-loading-dual-analytics-redundancy' retains Plausible as the sole analytics platform with no mention of consent gating (it describes Plausible as privacy-compliant and lightweight, implying it loads unconditionally). 'ux-analytics-triple-redundancy-kpi-corruption' says 'keep it deliberately as a privacy-first secondary source' without specifying whether it requires consent gating. Plausible's consent requirement depends on whether it sets cookies or processes personal data in the deployment configuration — this must be determined once and applied consistently across all proposals.

6. FONT PRELOAD HINTS AND CONSENT GATING ARE MUTUALLY CONTRADICTORY: Template proposal 'resource-loading-lcp-svg-hero' and platform proposals for font self-hosting both add <link rel='preload'> hints in <head> for font files. Platform consent proposals ('privacy-1-no-consent-mechanism', 'privacy-cookies-consent-banner-dark-pattern') gate font loading behind consent. A <link rel='preload'> in <head> fires unconditionally on parse — it cannot be consent-gated. If fonts are self-hosted (first-party), they do not require consent gating and preload hints are safe. If the team self-hosts fonts per the font proposals, the consent-gating of font loading in the consent proposals becomes unnecessary and the preload hints are correct. The proposals must be reconciled: self-hosted fonts = no consent gate needed = preload hints are safe.

7. SITEMAP TRAILING SLASH ALIGNMENT REQUIRES COORDINATION ACROSS THREE PROPOSALS: Template proposal 'prescan-escalation-3-canonical-trailing-slash' sets trailingSlash: 'always' in astro.config.mjs. Platform proposal 'ux-sitemap-unverifiable' installs @astrojs/sitemap and sets trailingSlash in Astro config. Template proposal 'escalation-og-twitter-meta-unverifiable' normalizes canonicalURL in SEO.astro to append trailing slashes. All three touch the same config value and the same canonical URL generation path. They must be implemented in the order: (1) astro.config.mjs trailingSlash setting, (2) sitemap integration, (3) SEO component normalization — and verified as a single coordinated deploy, not three independent ones.

8. BREADCRUMB JSON-LD AND CENTRALIZED SCHEMA ARCHITECTURE ARE UNCOORDINATED: Template proposal 'ux-nav-no-breadcrumbs' emits BreadcrumbList JSON-LD directly from the Breadcrumb.astro component via an inline <script type='application/ld+json'>. Template proposal 'escalation-unknown-schema-type' builds a SchemaGraph utility that composes all schema types into a single top-level @graph block per page. These two approaches are architecturally incompatible: the SchemaGraph pattern requires all schema to flow through a single serialization point, but the Breadcrumb component emits its own independent JSON-LD block. The Breadcrumb component must be refactored to return a schema object to the SchemaGraph rather than emitting its own <script> tag, or the SchemaGraph must explicitly consume the BreadcrumbList factory. This coordination must be specified before either is implemented.

9. AUTHOR SCHEMA AND CENTRALIZED SCHEMA ARCHITECTURE ARE UNCOORDINATED: Template proposal 'ux-revenue-2' adds Person JSON-LD for blog post authors directly in the BlogPosting template. Template proposal 'escalation-unknown-schema-type' centralizes all JSON-LD through SchemaGraph factories. Same conflict as breadcrumb above — the author Person entity must be emitted through the SchemaGraph BlogPosting factory, not as a separate inline block.

10. CONTACT PAGE TEMPLATE IS BEING MODIFIED BY FOUR INDEPENDENT PROPOSALS: 'ux-content-sparse-above-fold' restructures the contact page into a two-column layout. 'ux-conversion-no-trust-signals' injects a trust rail alongside the form. 'ux-conversion-cta-text-generic' changes the submit button text. 'ux-form-url-field-mobile-friction' removes the URL field. All four modify the same template file. If implemented in separate deploys without coordination, each deploy risks overwriting the previous one's changes. These must be sequenced: (1) URL field removal (reduces form complexity before restructuring), (2) button text change, (3) two-column layout restructure, (4) trust rail injection into the new layout.

## Within-Tier Interactions

Fixes within the same tier that share implementation surfaces (DOM manipulation, CSS properties, MutationObservers) and need coordinated rollout.

### Platform Tier

**Proposals:** `det-security-headers-https-weknowthewhy-com-about`, `privacy-1-no-consent-mechanism`, `det-gdpr-pre-consent-tracking-https-weknowthewhy-com`, `privacy-cookies-consent-banner-dark-pattern`, `ux-analytics-triple-redundancy-kpi-corruption`, `fonts-google-fonts-privacy-performance`, `resource-loading-dual-analytics-redundancy`, `ux-mobile-hamburger-discoverability`, `an-2-duplicate-analytics-gtm-ga4-plausible`, `mobile-nav-touch-targets-ux`, `resource-3-google-fonts-external-render-blocking-risk`, `js-4-ga4-collect-aborted-data-loss`, `js-unused-bytes-low-but-present`, `escalation-5-ga4-err-aborted-root-cause`, `ux-interactive-consent-banner-button-clarity`, `backwards-compat-astro-modern-css`, `navigation-no-search`, `attribution-split-pixel-not-applicable`, `ux-analytics-cross-domain-unknown`, `ux-sitemap-unverifiable`

**Conflicts:**

- CRITICAL ARCHITECTURAL CONFLICT — Analytics stack: Five proposals (ux-analytics-triple-redundancy-kpi-corruption, an-2-duplicate-analytics-gtm-ga4-plausible, js-4-ga4-collect-aborted-data-loss, js-unused-bytes-low-but-present, escalation-5-ga4-err-aborted-root-cause) all prescribe removing the standalone gtag.js and retaining GTM+GA4. One proposal (resource-loading-dual-analytics-redundancy) prescribes the opposite: remove GTM+GA4 entirely and retain Plausible as the sole analytics platform. One proposal (attribution-split-pixel-not-applicable) presents both options as a decision gate. These are mutually exclusive outcomes. Deploying any GTM-retention proposal after the Plausible-only proposal (or vice versa) produces a broken or double-instrumented state. A single architectural decision must be made before any of these proposals ships.
- CRITICAL CONSENT ARCHITECTURE CONFLICT — Three proposals (privacy-1-no-consent-mechanism, det-gdpr-pre-consent-tracking-https-weknowthewhy-com, privacy-cookies-consent-banner-dark-pattern) each implement a complete, independent consent manager with different storage mechanisms, different cookie/localStorage keys, different banner DOM structures, and different public APIs. privacy-1-no-consent-mechanism uses localStorage key 'mttr_consent' and exposes window.MttrConsent. det-gdpr-pre-consent-tracking uses a cookie named 'site_consent_v1' and no public API. privacy-cookies-consent-banner-dark-pattern uses a cookie named 'consent_state' and exposes window.ConsentGate. Deploying more than one of these produces duplicate banners, conflicting consent state reads, and a race condition where one manager injects scripts while another has recorded a decline. Exactly one consent implementation must be selected and the other two discarded.
- CONSENT + CSP SEQUENCING CONFLICT — det-security-headers-https-weknowthewhy-com-about deploys a nonce-based CSP via Netlify Edge Function that rewrites all <script> tags in the HTML response. All three consent manager proposals inject GTM dynamically via document.createElement('script') after consent is granted. Dynamically injected scripts do not pass through the Edge Function's regex rewriter and therefore carry no nonce. Under an enforced (non-report-only) CSP with 'strict-dynamic', these injected scripts will be blocked unless the nonce is passed to them explicitly. The CSP proposal's own rationale states that 'strict-dynamic' allows GTM's child scripts to inherit trust — but this only applies to scripts created by a nonced parent script. A dynamically injected GTM loader created by the consent manager's inline script (which itself has a nonce) should inherit trust via strict-dynamic, but this chain must be explicitly verified in staging. If the consent manager's inline script does not carry a nonce (because it is injected after the Edge Function has already processed the response), the entire chain breaks. The two systems must be tested together before either is enforced.
- GOOGLE FONTS DUAL PROPOSAL CONFLICT — Two proposals (fonts-google-fonts-privacy-performance and resource-3-google-fonts-external-render-blocking-risk) both prescribe self-hosting Google Fonts but with different implementation details: fonts-google-fonts-privacy-performance uses an @supports (font-variation-settings: normal) override pattern with static fallbacks declared unconditionally first; resource-3-google-fonts-external-render-blocking-risk uses format('woff2-variations') src descriptors and preloads both Inter and Lora unconditionally. The file naming conventions differ (inter-v4/Inter-roman.var.woff2 vs assets/fonts/v1/inter-latin-var.woff2). The @font-face src format strings differ ('woff2' vs 'woff2-variations'). Deploying both produces duplicate @font-face declarations for the same font-family names, which browsers resolve by specificity/order — the result is unpredictable rendering. One proposal must be selected. fonts-google-fonts-privacy-performance is preferred: it has the correct @supports structure (static unconditional baseline + variable override), uses format('woff2') which has broader support than format('woff2-variations'), and its revision_notes document the rationale for the structural choices.
- HAMBURGER BUTTON DOUBLE-FIX CONFLICT — Two proposals (ux-mobile-hamburger-discoverability and mobile-nav-touch-targets-ux) both target the hamburger button touch target and aria attributes. ux-mobile-hamburger-discoverability applies min-width/min-height: 48px via .mobile-nav-toggle. mobile-nav-touch-targets-ux applies the same via [data-hamburger], button[aria-label*='menu'], and .hamburger selectors. If both ship, the hamburger button receives duplicate CSS rules from different selectors, and the aria-expanded JS logic in ux-mobile-hamburger-discoverability may conflict with the global touch target CSS in mobile-nav-touch-targets-ux. The JS aria-expanded wiring in ux-mobile-hamburger-discoverability is the more complete fix (it handles label toggling and SVG hardening); the global touch target CSS in mobile-nav-touch-targets-ux is the more complete CSS fix (it covers all four component zones). These should be merged into a single implementation, not deployed independently.

**Interaction effects:**

- CONSENT → ANALYTICS LOAD ORDER: Whichever consent manager is selected must be deployed before or simultaneously with the analytics deduplication fix. If the analytics deduplication fix (removing standalone gtag.js) ships first without a consent gate, GTM still loads unconditionally on every page — the GDPR violation persists. The consent gate is a prerequisite for the analytics fix to be legally meaningful, not just technically correct.
- CONSENT → GOOGLE FONTS: All three consent manager proposals include self-hosting Google Fonts as a prerequisite step (Step 1 in privacy-1-no-consent-mechanism; Step 3 in privacy-cookies-consent-banner-dark-pattern). The Google Fonts self-hosting proposals (fonts-google-fonts-privacy-performance, resource-3-google-fonts-external-render-blocking-risk) are therefore a hard dependency of the consent architecture. Google Fonts must ship before or simultaneously with the consent manager — if the consent manager ships first and gates Google Fonts behind consent, users who decline will have no fonts until the self-hosting fix lands.
- CSP NONCE → CONSENT MANAGER INLINE SCRIPT: The CSP Edge Function injects nonces onto <script> tags present in the HTML response body. The consent manager is specified as an inline <script> in <head> in all three consent proposals. The Edge Function's injectNonce() regex will match this inline script and inject a nonce onto it — this is correct behavior. However, the consent manager's dynamically created scripts (GTM injection, Plausible injection) are created after the Edge Function has processed the response and will not have nonces. Under strict-dynamic, scripts created by a nonced script inherit trust. This means the consent manager's inline script (nonced by Edge Function) can create trusted child scripts (GTM, Plausible) without those children needing individual nonces. This chain is architecturally sound but must be validated in GTM Preview + CSP report-only mode before enforcement.
- CSP NONCE → FONT SELF-HOSTING: The CSP proposal's style-src directive includes 'self' and fonts.googleapis.com. After Google Fonts are self-hosted, fonts.googleapis.com can be removed from style-src and font-src. The CSP proposal's font-src includes 'self' and fonts.gstatic.com — after self-hosting, fonts.gstatic.com can be removed from font-src. These CSP directives should be tightened as part of the Google Fonts self-hosting deployment, not left as dead allowlist entries.
- ANALYTICS DEDUPLICATION → GA4 DASHBOARD ALERTS: All analytics deduplication proposals note that session counts will drop ~50% post-fix. If the cross-domain proposal (ux-analytics-cross-domain-unknown) ships simultaneously, the session count drop will be compounded by the removal of phantom sessions from the dual-beacon architecture. Any automated GA4 alerts, Looker Studio dashboards, or Google Ads Smart Bidding models must be updated before or immediately after deployment. Shipping these in the same release window without stakeholder preparation will trigger false-positive alerts.
- SITEMAP → CONSENT GATE: The sitemap validation script (ux-sitemap-unverifiable) uses HEAD requests to validate sitemap URLs. If the consent gate (any of the three consent proposals) renders the page differently for first-time visitors (e.g., blocking rendering until consent is given), the HEAD request from the CI validation script will not trigger the consent banner and will receive a 200 — this is correct behavior. No conflict, but the validation script should be confirmed to work against the post-consent-gate build output.
- BACKWARDS COMPAT GUARDRAIL → CONSENT MANAGER: The backwards-compat-astro-modern-css proposal adds an island-syntax-check Vite plugin that scans src/components for modern JS syntax. The consent manager (whichever is selected) is specified as an inline <script> block, not an Astro island component — it will not be scanned by the island-syntax-check plugin. This is correct: the consent manager uses ES5-compatible syntax (var, function declarations, no arrow functions in the production code paths) and does not need transpilation. No conflict, but the COMPATIBILITY.md should document that the consent manager is intentionally ES5 and exempt from the island syntax check.
- ESCALATION-5 FIRST-PARTY PROXY → CSP CONNECT-SRC: escalation-5-ga4-err-aborted-root-cause proposes a first-party Nginx proxy at /analytics/collect forwarding to google-analytics.com. If this ships alongside the CSP proposal, the CSP's connect-src directive must include the first-party proxy path (e.g., 'self' already covers same-origin paths, so /analytics/collect is implicitly allowed). However, the CSP proposal's connect-src currently lists https://www.google-analytics.com explicitly — after the proxy is deployed, this entry becomes unnecessary for the proxied path but may still be needed if any non-proxied GA4 requests exist. The connect-src should be audited post-proxy-deployment to remove the direct google-analytics.com entry if all beacons route through the proxy.
- NAVIGATION-NO-SEARCH PAGEFIND → ASTRO BUILD PIPELINE: The navigation-no-search proposal adds Pagefind as a post-build step ('npx pagefind --site <build-output-dir>'). The backwards-compat-astro-modern-css proposal adds a post-build CSS audit script. Both run after 'astro build' in CI. These are additive and non-conflicting, but the CI pipeline must sequence them correctly: astro build → CSS audit → Pagefind index → sitemap validation (sitemap validation must run last since it validates the fully built output including Pagefind assets).

**Recommended approach:** The 20 proposals collapse into 5 workstreams that should be sequenced as discrete releases, not deployed simultaneously:

RELEASE 1 — FOUNDATION (no user-facing changes, lowest risk): Deploy the security headers netlify.toml block (non-CSP headers only from det-security-headers). Deploy Google Fonts self-hosting using fonts-google-fonts-privacy-performance (the structurally superior proposal). Deploy the backwards-compat guardrail system. These three are independent, non-conflicting, and unblock everything else.

RELEASE 2 — CONSENT ARCHITECTURE (select exactly one implementation): Select privacy-cookies-consent-banner-dark-pattern as the consent manager — it is the most complete implementation (template-gating via <template> elements is the most robust pre-consent blocking mechanism, it has visual parity CSS, and it handles the revocation path). Discard privacy-1-no-consent-mechanism and det-gdpr-pre-consent-tracking as redundant. This release also removes all hardcoded GTM/GA4/Plausible script tags from HTML templates (prerequisite for both consent and analytics fixes).

RELEASE 3 — ANALYTICS ARCHITECTURE DECISION (requires client decision before coding): The client must choose between (A) GTM+GA4+Plausible with deduplication (ux-analytics-triple-redundancy-kpi-corruption path) or (B) Plausible-only (resource-loading-dual-analytics-redundancy path). This is a business decision, not a technical one. If GA4 is retained, the cross-domain linker work (ux-analytics-cross-domain-unknown) and conversion event instrumentation also ship in this release. If Plausible-only is chosen, the cross-domain and conversion work is replaced by Plausible custom events. The five redundant 'remove standalone gtag.js' proposals (an-2-duplicate-analytics-gtm-ga4-plausible, js-4-ga4-collect-aborted-data-loss, js-unused-bytes-low-but-present, escalation-5-ga4-err-aborted-root-cause, attribution-split-pixel-not-applicable) are all addressing the same root cause — pick one as the implementation reference and discard the others.

RELEASE 4 — CSP ENFORCEMENT: Deploy the nonce-based CSP Edge Function in report-only mode. Run for 48-72 hours. Validate that the consent manager's dynamically injected scripts are trusted via strict-dynamic chain. Switch to enforced mode. This must come after Release 2 (consent manager must be in place before CSP enforcement, or the consent manager's inline script may be blocked).

RELEASE 5 — UX AND DISCOVERY: Merge ux-mobile-hamburger-discoverability and mobile-nav-touch-targets-ux into a single implementation (one CSS file, one JS file). Deploy navigation-no-search (Pagefind + RelatedContent). Deploy ux-sitemap-unverifiable. Deploy ux-interactive-consent-banner-button-clarity as a refinement to the Release 2 consent banner if the visual parity was not fully addressed there.

### Server Tier

**Proposals:** `escalation-1-sri-gtm-ga-mitigation`, `server-transport-no-cdn-cache-headers`, `prescan-escalation-sri-gtm-incompatibility`, `st-6-cache-control-no-immutable-on-hashed-assets`, `server-transport-ttfb-excellent`, `cache-control-no-caching-static-assets`

**Conflicts:**

- CRITICAL — Triple cache-header duplication: 'server-transport-no-cdn-cache-headers', 'st-6-cache-control-no-immutable-on-hashed-assets', and 'cache-control-no-caching-static-assets' all produce a _headers file or netlify.toml [[headers]] block with identical /_astro/* immutable rules. All three will be assigned to the same developer against the same template. If implemented independently, the developer will create three separate PRs touching the same file, causing merge conflicts and unpredictable Netlify header precedence. Only one of these three proposals should be executed; the other two must be closed as duplicates before any work begins.
- CONFLICT — Netlify-CDN-Cache-Control vs X-Cache-Status header strategy: 'server-transport-no-cdn-cache-headers' adds 'Netlify-CDN-Cache-Control: public, max-age=31536000, immutable' as the CDN observability mechanism. 'cache-control-no-caching-static-assets' instead uses 'X-Cache-Status: {{x-cache}}' as the observability mechanism. These are different approaches to the same problem and cannot both be correct simultaneously. 'Netlify-CDN-Cache-Control' is a real Netlify directive that controls edge caching behavior; 'X-Cache-Status: {{x-cache}}' uses a Netlify header template variable syntax that is not documented for use in static _headers or netlify.toml files — it is only valid in Netlify Edge Functions response contexts. The '{{x-cache}}' variable in a static netlify.toml [[headers]] block will be emitted as the literal string '{{x-cache}}' rather than the cache status value, making it a broken observability mechanism.
- CONFLICT — Dual CSP proposals with divergent SRI stances: 'escalation-1-sri-gtm-ga-mitigation' (Phase 4) explicitly defers Plausible SRI and rejects Google Fonts SRI entirely, treating host allowlisting as sufficient. 'prescan-escalation-sri-gtm-incompatibility' (Steps 5–6) mandates SRI on both Plausible's script and the Google Fonts stylesheet as part of the same implementation. These two proposals cannot both be implemented — they produce contradictory integrity attributes on the same HTML elements. A developer following both would either apply SRI (per prescan proposal) or omit it (per escalation proposal) with no clear authority to resolve the contradiction.
- CONFLICT — CSP nonce injection target environment mismatch: 'escalation-1-sri-gtm-ga-mitigation' (Code Example 2) implements nonce generation as Node.js/Express middleware. 'prescan-escalation-sri-gtm-incompatibility' (Code Example 1) implements nonce generation as a Cloudflare Worker. The site is deployed on Netlify (confirmed by all cache-header proposals), not on a Node.js/Express server or Cloudflare Workers. Neither implementation is directly applicable without adaptation. The two proposals give the developer contradictory starting points for the same middleware layer, with no guidance on which runtime is authoritative for this site.
- CONFLICT — GTM script tag nonce injection approach: 'escalation-1-sri-gtm-ga-mitigation' (Code Example 3) adds the nonce as an attribute on the external GTM script tag ('src=https://www.googletagmanager.com/gtm.js'). 'prescan-escalation-sri-gtm-incompatibility' (Code Example 2) uses the classic GTM inline loader snippet (the IIFE that calls document.createElement) and propagates the nonce to the dynamically created script element via 'j.setAttribute(nonce, ...)'. These are structurally different GTM integration patterns — one uses a plain external script tag, the other uses the standard GTM IIFE. A developer cannot implement both simultaneously; they must choose one pattern, but neither proposal acknowledges the other exists.

**Interaction effects:**

- SEQUENCING DEPENDENCY — CSP proposals must follow cache-header proposal: The cache-header fix (whichever of the three duplicates survives) should be deployed first. It is a zero-risk, no-JS-touch configuration change. The CSP proposals (escalation-1 and prescan-escalation) require server-side nonce infrastructure, template changes, and a 5–7 day Report-Only observation window. Deploying cache headers first gives the team a clean, stable baseline before introducing the higher-complexity CSP work. Deploying CSP first risks a site-wide JS blackout that would obscure whether cache-header changes are working correctly.
- SEQUENCING DEPENDENCY — Dual-beacon elimination is a shared prerequisite: Both CSP proposals (escalation-1 Phase 1 and prescan-escalation Step 1) require removing the standalone gtag.js as a prerequisite before constructing the CSP allowlist. This is correct — the allowlist scope depends on the final script origin inventory. However, because both proposals describe this step independently, there is a risk the developer performs it twice (e.g., removes the tag, verifies in DebugView, then encounters the second proposal and re-audits unnecessarily). The deduplication of this step should be explicit in the implementation ticket.
- INTERACTION — 'server-transport-ttfb-excellent' netlify.toml baseline conflicts with cache-header proposals: The TTFB proposal (server-transport-ttfb-excellent) commits a netlify.toml with a [[headers]] block containing HSTS, X-Content-Type-Options, X-Frame-Options, and Referrer-Policy under '/*'. All three cache-header proposals also write to netlify.toml [[headers]] or _headers. If the TTFB proposal's netlify.toml is committed first and a cache-header proposal then adds a separate [[headers]] for '/*', Netlify will have two '/*' header blocks. Netlify merges multiple [[headers]] blocks for the same path by applying all matching rules — but if both blocks set Cache-Control for '/*', the last block in file order wins. This interaction is not flagged in any of the individual proposals.
- INTERACTION — CSP frame-ancestors directive conflicts with X-Frame-Options in TTFB proposal: 'server-transport-ttfb-excellent' sets 'X-Frame-Options: DENY' in its netlify.toml baseline. Both CSP proposals add 'frame-ancestors self' to the CSP, which takes precedence over X-Frame-Options in modern browsers. The two directives are not contradictory in effect (both restrict framing), but 'frame-ancestors self' is more permissive than 'DENY' — it allows the site to be framed by itself, while DENY allows no framing at all. If the intent is zero framing, the CSP proposals should use 'frame-ancestors none' to match the DENY posture, not 'frame-ancestors self'. This inconsistency will silently change the clickjacking protection posture when CSP is enforced.
- INTERACTION — Vary: Accept-Encoding header present in st-6 but absent in the other two cache proposals: 'st-6-cache-control-no-immutable-on-hashed-assets' adds 'Vary: Accept-Encoding' to the /_astro/* rule. 'server-transport-no-cdn-cache-headers' and 'cache-control-no-caching-static-assets' omit it. If the surviving cache proposal is one of the latter two, the Vary header will be absent, which is the correct behavior for Netlify's CDN (Netlify handles content negotiation at the edge and does not require Vary to serve correct encodings). However, the inconsistency across proposals means the developer has no clear signal on whether to include it. The st-6 proposal's rationale ('without Vary, a compressed response cached for one client may be served uncompressed to another') applies to intermediate proxies, not Netlify's own CDN, making it advisory rather than required.
- INTERACTION — Google Fonts SRI in prescan-escalation conflicts with CSP style-src in escalation-1: 'escalation-1-sri-gtm-ga-mitigation' sets 'style-src: self unsafe-inline https://fonts.googleapis.com' without SRI. 'prescan-escalation-sri-gtm-incompatibility' adds an integrity attribute to the Google Fonts stylesheet link tag. SRI on a stylesheet requires the response to pass CORS (crossorigin=anonymous) and the hash to match exactly. Google Fonts CSS is content-negotiated per User-Agent (different browsers receive different @font-face declarations). A hash computed in CI against one User-Agent will fail SRI validation for a different User-Agent, causing the stylesheet to be blocked silently. The escalation-1 proposal correctly identifies this and rejects Google Fonts SRI; the prescan proposal incorrectly mandates it. This is a functional correctness conflict, not just a style difference.

**Recommended approach:** Consolidate into a two-track implementation plan with a single authoritative file per concern:

TRACK A — Cache Headers (deploy first, this week): Close 'server-transport-no-cdn-cache-headers' and 'cache-control-no-caching-static-assets' as duplicates. Merge the TTFB proposal's security headers ([[headers]] for HSTS, X-Content-Type-Options, X-Frame-Options, Referrer-Policy) into the same netlify.toml as the cache rules — one file, one PR, no merge conflicts. Resolve the runtime conflict by writing a single Netlify-native nonce implementation: Netlify Edge Functions (not Cloudflare Workers, not Express) generate the nonce, inject it into HTML via HTMLRewriter, and set the CSP header. The GTM integration pattern should use the plain external script tag with nonce attribute (escalation-1 Code Example 3 pattern) rather than the IIFE pattern, since Astro's templating makes the external-src approach cleaner. ## Regression Risks

Potential regressions if fixes are implemented without the coordination described above, grouped by impact area.

### General

- CACHE IMMUTABILITY + CONSENT GATE SCRIPT INJECTION: Server-tier immutable cache headers (max-age=31536000, immutable) on /_astro/* assets are correct for content-hashed files. However, the platform-tier consent gate module (consent-gate.js) — if it lands under /_astro/* as part of the Astro build — will also receive immutable headers. If the consent gate logic is updated (e.g., to fix a GDPR compliance issue), users with the old consent-gate.js cached for up to a year will continue running the broken consent logic. Mitigation: ensure consent-gate.js is either (a) inlined directly into the HTML <head> as a <script> block (not a separate file, so it is never cached independently), or (b) served from a non-/_astro/* path without immutable headers. The inline approach is already implied by several consent proposals but must be explicitly enforced.
- NETLIFY EDGE FUNCTION (CSP NONCE) + IMMUTABLE CACHE HEADERS = CACHE POISONING RISK: The platform-tier CSP nonce Edge Function rewrites HTML responses to inject nonce attributes and the CSP header. If Netlify's CDN caches the HTML response (which it may do if Netlify-CDN-Cache-Control is set on /* routes), a cached response will contain a stale nonce that does not match the nonce in the CSP header of subsequent requests. This will cause all inline scripts to be blocked by CSP for users served the cached HTML. Mitigation: the Edge Function must set Cache-Control: no-store or Netlify-CDN-Cache-Control: no-store on all HTML responses it processes. The immutable cache headers must be strictly scoped to /_astro/* (hashed assets only) and must never apply to HTML documents. The server-tier proposals already specify this scoping, but the Edge Function must not inadvertently override it.
- TRAILING SLASH REDIRECT CHAIN + CANONICAL TAG MISMATCH DURING TRANSITION: Template proposal 'prescan-escalation-3-canonical-trailing-slash' changes astro.config.mjs trailingSlash to 'always', which changes all generated file paths. During the deploy window, Netlify's CDN may serve cached versions of pages with the old canonical URLs (without trailing slashes) while the new HTML contains trailing-slash canonicals. This creates a temporary canonicalization split that Googlebot may observe if it crawls during the transition. Mitigation: coordinate the trailing-slash config change with a cache purge on Netlify immediately after deploy. This is a short window but non-zero risk for a site with active crawl budget.
- FONT SELF-HOSTING + IMMUTABLE CACHE HEADERS + VERSION DIRECTORY: Server-tier proposals apply immutable cache headers to /_astro/* and optionally /fonts/*. Template proposal 'resource-loading-font-subsetting-opportunity' notes that font version strings are hardcoded in preload hints, @font-face src URLs, server config, and verification scripts. If a font file is updated (e.g., a new Inter release) but the version directory path is not changed, the immutable-cached old file will be served to returning users for up to a year. The proposals acknowledge this risk but do not specify a CI enforcement mechanism. Mitigation: the font version directory must be treated as a content-addressed path — any font file change must increment the version string, and a CI check must verify that the version string in the preload hint matches the version string in the @font-face src.
- CONSENT GATE BLOCKING GTM + FORM TRACKING PROPOSAL DEPENDENCY: Template proposal 'ux-analytics-form-submit-untracked' attaches a submit event listener and pushes to window.dataLayer, relying on GTM being loaded and a GA4 Event tag being configured. If the platform-tier consent gate blocks GTM from loading until consent is granted, and a user submits the contact form before granting consent, the dataLayer push will succeed (dataLayer is a plain array, always available) but GTM will not be listening — the generate_lead event will be silently lost. The proposal's belt-and-suspenders beforeunload listener has the same problem. Mitigation: the form tracking proposal must be updated to check ConsentGate.hasConsented() before pushing to dataLayer, or to queue events and replay them after GTM loads post-consent. This is a cross-tier dependency that neither proposal currently addresses.
- PAGEFIND SEARCH INDEX + CONSENT-GATED CONTENT: Platform proposal 'navigation-no-search' installs Pagefind, which crawls the built HTML and indexes all page content. If any page content is conditionally rendered based on consent state (e.g., personalized content injected post-consent), Pagefind will index the pre-consent version of those pages. For this static site with no personalized content, this is low risk currently. However, if future proposals add consent-conditional content rendering, the Pagefind index will not reflect it. Document this as a known constraint.
- SRCSET ADDITION + IMMUTABLE CACHE HEADERS: Template proposal 'resource-4-no-srcset-on-team-images' generates 2x WebP variants named with a @2x or -2x suffix. If these files land under /_astro/* (Astro processes them through its image pipeline), they receive immutable cache headers. If the headshot images are later updated (new team member photos), the content hash in the filename will change, which is correct behavior — old cached files become orphaned but are never served. This is safe. However, if the images are placed in /public/ with static filenames (no hash), they will receive immutable headers but the filename will not change on update, causing stale images to be served for up to a year. Mitigation: confirm all team images flow through Astro's image pipeline (which adds content hashes) before applying immutable headers to their path.
- SKIP LINK ADDITION + CONSENT BANNER FOCUS TRAP: Template proposal 'det-wcag-improper-content-structure-https-weknowthewhy-com-about' and 'a11y-5-mobile-menu-duplicate-nav-no-aria-label' both add a skip-to-main-content link as the first focusable element in <body>. If the consent banner is present on page load and traps focus, the skip link — which is the first focusable element in DOM order — will be unreachable until the banner is dismissed. This is correct ARIA dialog behavior (focus should be trapped inside the modal), but it means keyboard users cannot skip to main content until they interact with the consent banner.
